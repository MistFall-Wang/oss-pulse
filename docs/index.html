05The deliberate incident drill

Sprint 5b: synthesize 200 PushEvent rows where payload.size was renamed to payload.commit_count, ingest as a new ingest_hour, and watch where each gate in the pipeline actually catches it. The point of the drill is not the bug — it's discovering where the detection happens, and what's silently poisoned in between.

Detection chain — what actually fired (and what didn't)

1
bronze_ingest
Bronze stores payload as raw JSON STRING (ADR-0001). The rename is invisible at this layer — by design.
pass
2
gate_bronze (4 DQ checks)
id-uniqueness, type-in-set, public-flag, created_at-not-null all still hold; nothing checks payload contents at Bronze.
pass
3
silver build — events_push
get_json_object(payload_raw, '$.size') silently returns NULL on the 200 rows. The SELECT projects NULL successfully.
pass
4
gate_silver (row-count match)
Silver row count still equals Bronze filtered by type — NULLs count as rows. The contents-check this gate was missing is added below.
pass
5
dbt test — not_null_events_push_commit_size
Got 200 results, configured to fail if != 0. Detected — but only after Silver finished and (in a real DAG) Gold marts had already consumed the bad data.
fail

Root cause and what changed postmortem 0001

It's a gate-placement problem, not a missing-test problem. The dbt schema test does catch it — but it runs at the end of the pipeline, after Gold is already poisoned. The fix was two-part: a one-line coalesce(size, commit_count) in events_push.sql, plus a new silver_commit_size_not_null gate in quality/checks.py that moves detection between Silver and Gold, where it belongs. Every postmortem must leave behind a regression check — otherwise it's theatre.

Full 5 Whys and lessons in docs/postmortems/0001-schema-drift.md.

06Performance — and the honest negative result

Hypothesis: OPTIMIZE … ZORDER BY (type) on Bronze would prune per-type filter reads in every Silver build. Result: it didn't. The wall-clock improvement that showed up at first was JIT warmup, not data-skipping. Recording the finding honestly is what makes the experiment defensible at interview.

Root cause for the no-op result: our Bronze has 4 files (one per ingest_hour). ZORDER can re-arrange rows within a file but can't split below the file granularity — so the smallest skip-unit was 25 % of the table. At 100× scale, this experiment is worth repeating. The doubling-then-VACUUM finding is the empirical reason ADR-0009 (compact-daily, vacuum-weekly with 168h retention) goes from aspirational to mandatory.

07The seven senior signals — where each one lives

The project's north star: every Sprint reinforces at least one of these. Each card below names the artifact that proves it, so an interviewer can ask "show me" instead of trusting the claim.

01Idempotency
Bronze + Silver + Gold all MERGE on natural ids. Cross-layer invariant count(*) == count(distinct id) enforced by 3 runtime spark/jobs/gold_*_verify.py scripts.
02Backfill / replay
Airflow DAG params.start_hour / end_hour. docs/runbooks/backfill.md documents Airflow and shell-only invocation paths.
03Schema-drift tolerance
ADR-0001 puts payload as raw JSON STRING in Bronze, then ADR-0005 has Silver parse on demand. postmortems/0001-schema-drift.md is the empirical proof under a real injected break.
04Data-quality gates
18 checks in quality/runner.py across 4 suites; each suite exits non-zero to gate the next Airflow task. Lightweight on purpose — design trade-off vs full GE documented in checks.py.
05Perf tuning report
5-dimension before/after measurement with an honest negative result and a per-finding follow-up plan in docs/performance/sprint5b_tuning.md.
06Batch + streaming story
Redpanda + replay + Structured Streaming + reconcile script in streaming/. 181,221 events reconciled to zero delta.
07Operational docs
7 accepted ADRs + 5 runbooks + 1 postmortem + 3 mart design docs + 1 spike report. Every decision that was made on purpose has a paper trail.

09Sprint timeline

Original 6-week estimate revised to a realistic 8–10 weeks. Streaming was de-scoped from a 4-week build to a 1-week MVP that ships the talking point.

Sprint 0Schema discovery + ADR-0001

Cross-year sample (2015 / 2018 / 2025): top-level envelope stable for 10 years; IssueCommentEvent nested paths grew 120 → 309. Drives Bronze payload-handling decision.

Sprint 1Bronze ingestion + first Silver

PySpark + Delta MERGE on event_id; first Silver model events_push via dbt-spark; ADR-0002 and ADR-0003 codified. 613,876 Bronze rows, 0 duplicates.

Sprint 2 + 2.5First Gold mart + bot-heuristic spike

repo_daily_activity (162,719 rows, composite-key MERGE). Spike proved the original Rule B (payload.performed_via_github_app) was at the wrong path; would have broken Sprint 3 if not caught.

Sprint 32 more marts + 5 more Silver tables

oss_health_mart + bot_vs_human_activity_mart; ADR-0006 codifies Rule A + Rule C + is_app_event; is_bot() macro centralises classification; cross-mart verifier promoted to a quality gate.

Sprint 4DQ gates + Airflow + runbooks

Lightweight quality/ framework (intentionally not full GE — trade-off documented); parameterized Airflow DAG; runbooks for backfill, schema-change, data-missing, and airflow-setup.

Sprint 5bCI + perf report + ADR-0007 + postmortem-0001

GitHub Actions on PR (ruff, pytest, dbt parse). Deliberately-induced schema break, full detection-chain documentation, regression check added to silver gate.

Sprint 5aTerraform + cloud-migration runbook code-complete

S3 (3 buckets, KMS-encrypted, lifecycle-tiered) + cross-account IAM for Databricks workers. Apply requires AWS account + Databricks Free Edition signup (user-side step).

Sprint 6Streaming MVP

Redpanda + replay.py + Structured Streaming consumer + reconcile.py. 181,221 events end-to-end with 0 row delta.

Sprint 7-9Streaming production-grade backlog

Time-warped replay, watermark-based late-event handling, continuous reconciliation, ADR-0008 + 0009, real-dollar cost report. Triggered only by job-market signal.

10Run it locally

Requires JDK 17 (Spark 3.5 / Hadoop 3.3.4 break on Java 18+ due to Subject.getSubject removal). Tested on Amazon Corretto 17.

# Setup
git clone https://github.com/MistFall-Wang/oss-pulse && cd oss-pulse
uv sync
export JAVA_HOME=/Library/Java/JavaVirtualMachines/amazon-corretto-17.jdk/Contents/Home
export PATH="$JAVA_HOME/bin:$PATH"
export PYSPARK_SUBMIT_ARGS="--driver-memory 4g pyspark-shell"

# End-to-end on one ingest hour
uv run python -m spark.jobs.bronze_ingest --source data/raw/2025-01-15-12.json.gz --bronze-path data/bronze/events
uv run python -m quality.runner --layer bronze
cd dbt && uv run dbt deps && uv run dbt run --select silver && cd ..
uv run python -m quality.runner --layer silver
cd dbt && uv run dbt run --select gold && cd ..
uv run python -m quality.runner --layer gold
uv run python -m quality.runner --layer cross_mart
cd dbt && uv run dbt test

# Cross-layer verifiers (ground truth on the busiest rows)
uv run python -m spark.jobs.gold_verify
uv run python -m spark.jobs.gold_health_verify
uv run python -m spark.jobs.gold_bot_verify

# Streaming MVP
docker-compose -f streaming/docker-compose.yml up -d
uv run python -m streaming.replay --source data/raw/2025-01-15-12.json.gz
uv run python -m streaming.consumer
uv run python -m streaming.reconcile --ingest-hour 2025-01-15-12

# Deliberate incident drill (Sprint 5b)
uv run python -m spark.jobs.incident_inject
# ... observe gates ...
uv run python -m spark.jobs.incident_inject --cleanup
